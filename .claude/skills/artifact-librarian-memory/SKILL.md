# artifact-librarian memory

Behavioral memory only (DEC-0331): task patterns, distillation
approaches that worked, common refusals and their causes, stakeholder
preferences. NEVER corpus facts — corpus truth lives in the artifacts;
an entry contradicting the corpus is a bug to fix here.

Curation rules (for the artifact-librarian, the sole writer of this
file): update via Bash only, as part of finishing a task; keep the
Entries section at or under 100 lines — consolidate or drop the least
useful entries when near the cap; one entry per line, most broadly
useful first; date entries (YYYY-MM-DD). This file is the ONLY thing
you may write outside the typed write API.

## Entries

- 2026-07-09: two gw.py copies can exist (global skill dir vs project-local `.claude/skills/artifact-interact`); if the project has an uncommitted fix to gw_write.py (check `git status`/`git diff` there first), use the project-local copy — the global one may be stale.
- 2026-07-09: `write apply` batches are NOT fully atomic w.r.t. forward references — `Corpus.recheck` (post-write) is deferred across the whole batch, but `resolve_all` (pre-write validation inside edit-section/update-overview/create body content) is NOT deferred, so an op referencing an ID created later in the same batch fails immediately with "unresolved IDs" even though the batch as a whole would resolve. Fix: order ops so every ID is created before anything references it, or split into two apply calls (first the creates, then edits that reference them). This is a real gap vs DEC-0314's as-a-unit intent — report it, don't paper over it silently.
- 2026-07-09: `write create` does not YAML-quote titles containing `:` or `'` — a title like "... (dogfooding: the method's ...)" writes an unparseable frontmatter (`mapping values are not allowed here`), which then cascades into "does not resolve" failures on every other artifact that cites that ID. Caught only by `check`, not by the create op's own re-check. This is a create-op defect with no sanctioned repair op (no title/frontmatter-edit exists in the write API) — refuse-and-report to the caller, do not freehand-patch the .md file.
- 2026-07-09: after a batch apply partially fails (e.g. forward-reference ordering, see above), check `git status` before retrying — untracked newly-created files can be safely removed and re-attempted, but do NOT re-run already-succeeded `create` ops (they'd allocate new duplicate IDs); instead build a remaining-ops batch of only the unexecuted ops, reordered as needed.
- 2026-07-09: `search audit` (and other `search`/`graph` families, run via `uv`) print progress bars and warnings to stderr — when redirecting a family's output to a saved JSON packet, redirect stdout only (`> file.json`, not `> file.json 2>&1`) or the file becomes invalid JSON.

Post-write check re-confirmed it independently either way, but a pre-flight grep would have surfaced the discrepancy before spending a write op.

- 2026-07-10: `edit-section`'s `section_span` always resolves to the FIRST heading matching the given text and stops at the very NEXT heading of level<=that, regardless of that next heading's text — so on a body with two consecutive identical headings (a corruption from a prior edit-section call that mistakenly included the heading line in its content), edit-section can only ever touch the span between the two duplicates (often empty), never reach or remove the second/real one. There is no delete-section, merge-section, or occurrence-index flag. Do not "fix" this by writing content into the first span — that duplicates the body instead. The only sanctioned recovery is to revert that span back to empty (restores the pre-existing single-content state) and refuse-and-report the leftover spurious empty heading as an API gap; do not freehand-strip the extra heading line even though it looks trivial.
- 2026-07-10: for a stakeholder-authorized one-off manual out-of-band edit (outside the write API), verify with `read outline` (heading appears once, etc.) then run `check` — do not re-derive correctness from the diff yourself; the checker's verdict line is the report.
- 2026-07-10: confirmed recurring — code-span-wrapped IDs (e.g. `EP-0009`) reliably trip write_apply's post-write recheck as "unresolved" false positives while the full `check` gate (code-span-aware) passes clean; when a caller flags this in advance and pre-flight grep confirms the source content matches what they described, treat the apply-time refusal as expected noise, still finish with `check` as the authoritative verdict, and report both explicitly.
- 2026-07-10: the code-span carve-out (SPEC-artifact-common rule 7) is body-Markdown-scoped only — a bare ID wrapped in backticks inside the *frontmatter overview field* (not body) still trips BOTH write_apply's recheck AND the full `check` gate as unresolved (FAIL, not just a recheck false-positive); this differs from the body-context case (prior 07-10 entry) where full `check` correctly passes. Do not assume backtick-wrapping an intentionally-unresolvable ID anywhere in a create/write op's --overview text will satisfy the checker — if the task's literal requested overview text embeds such an ID, attempt it once, and if `check` itself (not just recheck) fails, stop and refuse-and-report rather than paraphrasing the ID away yourself; that rewording choice belongs to the caller.
- 2026-07-10: `write create --from-file BODY` uses BODY verbatim (no auto H1/heading substitution) — unlike the templated-body path (no --from-file), which writes `# {aid}: {title}` itself. When drafting a body file in advance, either omit the H1 line (create does NOT require one — checker cares about section headings, not the H1) or pre-derive the next sequential ID (`ls docs/<type-dir>/ | sort` for the highest existing N) so the H1 you author matches the ID that will actually be allocated.
